§ 01
Introduction
Purpose, scope, definitions, and document overview
Purpose
This SRS defines all requirements for Smart Thesis Hub (STH) — a web and mobile-friendly platform that replaces manual, fragmented thesis management processes in universities with a unified digital system. This document serves as the foundation for design, development, testing, and maintenance.

1.1 Scope: STH digitizes the entire thesis workflow including student registration, supervisor discovery, project browsing, request management, real-time notifications, video resources, and administrative oversight. The system serves three primary user roles: Student, Supervisor, and Admin (System Administrator).

1.2 Problem Statement: Currently, students must navigate Google Sites, email threads, or physical notice boards to find supervisors. There is no centralized system for submitting requests, tracking their status, or matching students to the best supervisors based on research interests. Smart Thesis Hub solves all of these pain points.

§ 13
Security & Ethics
How the system protects users and respects their data
🔒
Authentication
bcrypt hashing (12 rounds), JWT with 24h expiry and refresh tokens, 5 failed attempts triggers lockout
👮
Authorization
RBAC enforced at every API route. Students cannot access supervisor or admin endpoints. Middleware validates role on each request.
🛡️
Input Security
All inputs sanitized via Joi schema. XSS protection via Helmet.js. CSRF tokens on state-changing operations. Rate limiting on all endpoints.
🔐
Data Encryption
AES-256 encryption at rest (MongoDB Atlas). TLS 1.3 in transit. Sensitive fields never logged or exposed in responses.
Ethical Commitments
No user data is sold or shared with third parties · AI recommendations are explainable and transparent · Feedback is anonymous by default · Users can request full data export or permanent deletion at any time · GDPR Article 17 (Right to Erasure) fully supported
§ 14
Limitations & Challenges
Known technical and real-world challenges with mitigation strategies
🤖
AI Recommendation — Cold Start Problem
Challenge: New students with sparse profiles receive low-quality recommendations. Mitigation: Use department and research area as baseline matching signals. Improve accuracy as more interaction data accumulates over time.
⚡
Real-Time at Scale — WebSocket Limits
Challenge: Socket.io connections may become strained under very high concurrency (5,000+ users). Mitigation: Redis Pub/Sub adapter for Socket.io planned for Phase 2 to enable multi-server scaling.
🔗
University ERP Integration Dependency
Challenge: Integration relies on external API availability from the university's IT department. Delays in API access can block student verification. Mitigation: Manual admin verification fallback available.
📱
No Native Mobile App (Phase 1)
Challenge: The responsive web app lacks offline capability. Push notifications require the browser to be open. Mitigation: React Native app with offline sync and push notifications is scoped for Phase 2.
🎬
Video Hosting Bandwidth Costs
Challenge: AWS S3 bandwidth costs will increase significantly at scale. Mitigation: CloudFront CDN integration reduces origin costs by 70–80%. Video compression pipeline added to upload workflow.
🏫
Single-Tenant Architecture
Challenge: Current architecture is designed for one university. Multi-university SaaS deployment requires tenant isolation in the schema. Mitigation: Multi-tenancy via organizationId field is architected for Phase 3.
